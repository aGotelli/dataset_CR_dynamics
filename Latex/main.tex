\begin{abstract}

    This data descriptor presents a multi-sensor dataset on the quasi-static and dynamic motion of a tendon-driven continuum robot, including scenarios with and without contact with objects in the workspace.
    The robot is instrumented with a motion capture system, fiber Bragg grating sensors, cable tension gauges, motor encoders and a base force-torque sensor while the contact object is provided with a motion capture frame and a force-torque sensor.
    The dataset covers planar, circular, and Lissajous trajectories at multiple speeds.
    Both raw sensor recordings and processed data (filtered, time-aligned, uniformly resampled) are provided.
    The dataset enables benchmarking of dynamics models, training of data-driven methods, study of contact interactions, and sensor fusion research for continuum robots.
\end{abstract}

Continuum robots are manipulators whose body undergoes continuous elastic deformation, actuated by tendons, pneumatics, or other distributed mechanisms~\cite{Robinson1999, BurgnerKahrs2015}.
Their dynamics have been studied through various modeling approaches \cite{Renda2016,Till2019, Boyer2021, Renda2024}, and there is growing interest in applying machine learning methods, in both statics \cite{Bensch2024} and dynamics \cite{Du2026}, as well as studying their contact and interaction with the environment \cite{Rao2024} with new studies addressing the contact kinematics with differentiable properties \cite{Mathew2026}.
%
However, experimental validation in continuum robotics typically relies on ad-hoc prototypes instrumented with lab-specific sensors, and the resulting data is rarely made publicly available.
A related effort produced a dataset for concentric tube robots~\cite{Grassmann2022}, but no publicly available dataset currently combines kinematics and force measurements for a tendon-driven continuum robot undergoing both quasi-static and dynamic motions, including contact with objects in the workspace.
%
This limited availability of open datasets prevents reproducibility of results and the comparison of competing modeling, control, and learning approaches on common ground.


% --- dataset overview ---
This data descriptor presents a multi-sensor dataset recorded on a single-segment tendon-driven continuum robot.
The robot consists of an elastic backbone mounted with 3D printed disks providing guidance for the tendon-driven actuation.
%
Kinematic measurements are provided by an OptiTrack motion capture system (disk poses), FBGS (distributed strain and shape), motor encoders (tendon displacements), and IMUs (angular velocities and linear accelerations). 
Force measurements are provided by Mark10 gauges (cable tensions), an ATI force-torque sensor (base wrench), and a Resense force-torque sensor (contact forces).
%
The dataset is organized in three subsets:
(i) quasi-static motions,
(ii) dynamic motions,
and (iii) dynamic motions with contact of an object in the workspace.
Trajectories for dynamic motions include planar sweeps along x, y, star patterns, circular paths, and Lissajous figures, each executed with two maximum angular velocities for the actuators: slow ($1$ \si{\radian\per\second}) and fast ($3$ \si{\radian\per\second}).
Both raw sensor recordings and processed (filtered, resampled, time-aligned) data are provided.

% --- reuse value ---
The dataset is intended for benchmarking dynamics models against experimental data, training and evaluating data-driven methods, studying contact and interaction forces, investigating sensor fusion approaches, and performing parameter identification.
Because the robot is built from commercially available components and the design is fully documented, the dataset can also serve as a reference for groups building similar platforms.

In total, the dataset comprises N experiments totaling approximately X hours of multi-sensor recording (Y GB of raw data, Z GB processed).

\section{Methods}

In this section, we define the methodology used to create the presented dataset, starting from the robot design and components to the specifics of the sensors.
Finally, we describe how the data is collected giving a general idea of the data collection pipeline.



\subsection{Design}

In this section we describe both the design of the continuum robot and the contact wand that is used to interact with the robot.

\subsubsection{Continuum robot}
The overall objective of the design is to maximize modeling and sensing accuracy by minimizing unmodeled effects such as friction, structural compliance, and geometric uncertainty. All design choices aim to ensure close consistency between the physical system and the underlying continuum robot model. The system consists of a bench-top experimental platform and a tendon-driven continuum robot (Fig.~\ref{fig:robotPlatform}).

The experimental platform provides a rigid and geometrically stable structure for actuation, sensing, and robot mounting. It consists of a 3D-printed base platform (PETG), aluminum extrusion frames (\SI{40}{\milli\meter} T-slot profiles), a metal mounting plate (not shown in Fig.~\ref{fig:robotPlatform}), and additional 3D-printed mounting components. The base platform is rigidly attached to the frame, which supports the actuation and sensing units, and is mounted on a leveled metal plate with adjustable feet. This ensures that the direction of gravity is well-defined and consistent with the model. Actuation components are arranged symmetrically about the robot centerline, minimizing unintended momentum and maintaining base stability. 

The robot consists of a slender NiTi tubular backbone, allowing internal sensor integration. 
The material properties are given in Table~\ref{tab:robot parameter}.
Nine equally spaced spacer disks (\SI{6}{\centi\meter} spacing) are mounted along the backbone using set screws. 
The disks are fabricated from fiber-reinforced PLA, with weight-reducing cutouts while maintaining stiffness, and include mounting points for external markers. 
Each disk contains eight radially distributed holes for tendon routing. 
In the current configuration, four tendons are attached at the distal disk and routed proximally through the disks, remaining approximately parallel to the backbone with a constant radial offset of \SI{37.5}{\milli\meter} from the centerline.

At the base, tendons are redirected using 3D-printed pulley adapters mounted on the sensing interface of the force gauges, and then connected to the actuators. 
No intermediate pulleys are used between the robot and the force gauges, ensuring that the measured force directly corresponds to the applied tendon tension. 
The ATI force--torque sensor is mounted between the base platform and the robot base disk, providing a rigid interface for measuring the resultant forces and moments at the boundary. The base disk is reinforced to ensure a stiff connection, enabling accurate transmission of loads without additional compliance.

Overall, the design emphasizes structural rigidity, symmetric actuation, friction-minimized tendon routing, and well-defined boundary conditions, ensuring that the experimental system faithfully reflects the assumptions of the continuum model.

\subsubsection{Contact wand}
The contact wand is designed to provide a lightweight slender object capable of interacting with the robot without occluding the disks markers.
The wand is made of an aluminum extrusion frame (\SI{20}{\milli\meter} T-slot profile) provided with a mocap frame (to measure its pose in the robot workspace) and a force/torque sensor to measure the interaction force.
For the latter, an ad-hoc support has been realized to align the sensors axis to those of the mocap frame.

\subsection{Sensors}

Different sensors have been mounted on the robot to capture different aspect of the motion, being it kinematics like position, shape of the robot and pulled lenght of the cables, or force measures like tendon tensions, forces and torques at the base and at the contact with an object in the environment.
%
In this section, we describe the characteristics of each sensor in terms of frequency readings, range of outputs, and a summary of their specifics available on their datasheet.

\subsubsection{Mark 10}


Cable tension was measured using Mark-10 M3-5 digital force gauges (Mark-10 Corporation, Copiague, NY, USA; \url{https://mrmmetrology.com/product/m3-5-mark-10-digital-force-gauge/}), with a full-scale capacity of \SI{5}{lbf} ($\approx$\,\SI{22.2}{\newton}), a resolution of \SI{0.005}{lbf} ($\approx$\,\SI{0.02}{\newton}), and a manufacturer-specified accuracy of $\pm\,0.3\%$ of full scale.
All units were calibrated by an ISO~17025 accredited laboratory
with an expanded uncertainty of $\pm$\,\SI{0.007}{lbf} ($k{=}2$, 95\% coverage).
Each gauge was interfaced to the acquisition computer via a serial connection
at \SI{115200}{baud} and polled continuously using a request--response protocol.
The effective sampling rate, limited by the serial round-trip latency, was
approximately \SI{335}{\hertz} per gauge.
The output was recorded directly in newtons.
A dedicated Python script (one instance per gauge) opens the serial port via \texttt{pyserial},
sends an ASCII query command, parses the returned numeric string, and timestamps each reading with the system wall clock (\texttt{time.time()}).
All samples are buffered in a pre-allocated NumPy array and written to a CSV file at the end
of the recording.



\subsubsection{FBGS}
\label{sec:fbgs}


Distributed strain along the robot backbone is measured with an FBGS four-core multicore fiber
(MCF, ID: F13421-0007-003; FBGS International, Belgium) featuring 26 draw-tower gratings per
core at \SI{20}{\milli\meter} spacing, for an active sensing length of \SI{500}{\milli\meter}.
Three cores are used for shape reconstruction (core~2 omitted), enabling independent estimation
of bending curvature, bending direction, and temperature at each grating.

The fiber is interrogated by an FBG-Scan~904 spectrometer (\SIrange{1510}{1590}{\nano\meter},
4 channels, wavelength precision \SI{0.3}{\pico\meter} $1\sigma$).
ILLumiSense Shape~CORE~v1.1 performs peak detection and outputs the reconstructed 3-D shape
at \SI{1}{\milli\meter} arc-length resolution and \SI{100}{\hertz}.
A C++ client receives the TCP stream, timestamps each sample with
\texttt{std::chrono::high\_resolution\_clock}, and writes the buffered data to CSV after
the recording.

Prior to use, the distal grating was located to $\sim$\,\SI{1}{\milli\meter} via a hot-wire contact procedure and aligned with the tip disk; the fiber is fixed at the base and tip to avoid twisting inside the backbone.
% A residual angular offset between the fiber and robot bending planes is corrected
% in post-processing using the initial planar calibration sweeps
% (see Section~\ref{sec:kin_cross_val}).


\subsubsection{ATI-FT}

External force and torque measurements at the robot base were acquired using an ATI Mini40
six-axis force/torque sensor (calibration SI-40-2; ATI Industrial Automation, Apex, NC, USA;
\url{https://www.ati-ia.com/products/ft/ft_models.aspx?id=mini40}).
% The Mini40 is a monolithic transducer machined from hardened stainless steel via electrical
% discharge machining (EDM), instrumented with silicon strain gages that provide a signal-to-noise
% ratio approximately 75 times higher than conventional foil gages.
% The transducer has a diameter of \SI{40}{\milli\meter}, a height of \SI{12.2}{\milli\meter},
% and a mass of \SI{49.9}{\gram}.
With the SI-40-2 calibration, the sensing ranges are
$\pm\SI{40}{\newton}$ in $F_x$ and $F_y$,
$\pm\SI{120}{\newton}$ in $F_z$,
and $\pm\SI{2}{\newton\meter}$ in $T_x$, $T_y$, and $T_z$.
% TODO: verify resolution values from your calibration certificate
Single-axis overload ratings are $\pm\SI{810}{\newton}$ ($F_{xy}$) and
$\pm\SI{2400}{\newton}$ ($F_z$), and the resonant frequencies are
\SI{3200}{\hertz} ($F_x$, $F_y$, $T_z$) and \SI{4900}{\hertz} ($F_z$, $T_x$, $T_y$).
The transducer was connected in the DAQ~F/T configuration, with the six analog voltage channels
($\pm\SI{10}{\volt}$ range) digitized by a National Instruments data acquisition card
using the \texttt{nidaqmx} Python driver.
The hardware sampling clock was set to \SI{1000}{\hertz}; every five consecutive samples
per channel were averaged to yield an effective wrench output rate of approximately \SI{200}{\hertz}.
A $6\times6$ calibration matrix provided by ATI was applied in software to convert the
bias-subtracted voltage readings into calibrated force and torque values
($F_x$, $F_y$, $F_z$, $T_x$, $T_y$, $T_z$).

\subsubsection{Actuators}

The tendons are actuated by four brushless DC motors (Cybergear, Xiaomi, China) with custom 3D-printed tendon spools of winding radius \SI{20}{\milli\metre}, i.e., the radius at which the tendon cable is coiled around the spool. 
Each motor is a quasi-direct-drive (QDD) module integrating a three-phase, 28-pole frameless torque motor with a 7.75:1 planetary gearbox, a 14-bit single-turn absolute encoder, and onboard FOC electronics, in a compact package of \SI{80.5}{\milli\metre} diameter and \SI{36.5}{\milli\metre} height (\SI{317}{\gram}). 
The motors are powered at \SI{24}{\volt} DC and provide a continuous output torque of 
\SI{4}{\newton\metre} and a peak torque of \SI{12}{\newton\metre}. 
Communication with the host PC is handled via a CAN 2.0 bus at \SI{1}{\mega\bit\per\second}. 
The four motors are operated as two antagonistic pairs (motors 1\&3 and 2\&4), each pair controlling one planar degree of freedom of the robot tip. 
They are operated in a closed-loop position control scheme, where target shaft angles 
$\theta^*$ are commanded directly over CAN, with shaft angle, velocity, and output torque returned as real-time feedback at each control cycle. 
Angles are computed relative to the home position, which is initialized as the reading angle at the motor startup.
%The resulting tendon displacement $\Delta l = r_s \, \theta^*$, where $r_s = \SI{20}{\milli\metre}$ is the spool winding radius, is a geometric byproduct of the motor rotation.
The CAN bus is accessed from the host PC through a USB--CAN adapter (Canable / \texttt{candle} interface) using the \texttt{python-can} library.
A custom Python script, built on the open-source Cybergear driver~\cite{CybergearDocs}, instantiates four \texttt{CANMotorController} objects, sets each motor to position mode, and commands target angles in a sequential loop.
At each control cycle the script reads back the shaft angle, velocity, and torque from every motor and timestamps the sample with the system wall clock (\texttt{time.time()}).
The feedback data is buffered in memory and saved to a single CSV file at the end of the recording.

\subsubsection{Motion Capture System}


The shape of the robot backbone and the pose of objects in contact with it are measured using an optical motion capture system (OptiTrack, NaturalPoint, USA) composed of five PrimeX13 and two PrimeX13W cameras. 
The PrimeX13 cameras feature a 1.3\,MP sensor ($1280 \times 1024$\,px), a native frame rate of \SI{240}{\hertz}, a field of view of \ang{56} $\times$ \ang{46}, and a 3D positional accuracy of 
$\pm$\SI{0.15}{\milli\metre}. 
The two PrimeX13W cameras share the same sensor and frame rate but provide a wider field of view of \ang{82} $\times$ \ang{70}, suited to ensure full coverage of the compact workspace, at a slightly reduced accuracy of $\pm$\SI{0.30}{\milli\metre}. 
All cameras use \SI{850}{\nano\metre} infrared illumination and communicate via GigE/PoE. 
Five rigid bodies composed of passive retro-reflective markers (\SI{6}{\milli\metre} diameter) are attached to equally-spaced disks along the robot backbone at arc-length positions $s \in \{0, 12, 24, 36, 48\}$\,cm, providing discrete samples of the backbone pose. 
An additional rigid body is mounted on the contact object to track its 6-DoF pose within the workspace. The system runs on a dedicated host PC managed by Motive~3.4, which processes and solves 3D pose at \SI{120}{\hertz}. 
Data are streamed in real time to the dataset acquisition PC via a point-to-point Ethernet link 
using the NatNet~4.4 SDK in unicast UDP mode. 
A custom Python client receives each frame and logs the 6-DoF pose --- position $(x, y, z)$ and orientation $(q_x, q_y, q_z, q_w)$ --- of each rigid body, together with a per-body tracking validity flag. 
Timestamps are assigned on the acquisition PC using the system wall clock (\texttt{time.time()}), rather than Motive's internal camera exposure timestamps, which could induce a small and variable 
network transmission jitter in the logged timing.



\subsubsection{Resense FT}

Contact forces between the robot and objects in the workspace were measured using a
Resense HEX\,12 six-axis force/torque sensor (ordering code HEX125S-020S-1R1-1; Resense GmbH, Klingenberg, Germany; \url{https://www.resense.io}).
% The HEX\,12 is a compact transducer machined from Titanium Grade~5, instrumented with foil strain gauges, with a diameter of \SI{12.5}{\milli\meter}, a height of \SI{11.0}{\milli\meter},
% and a mass of \SI{3.3}{\gram}.
The nominal measurement ranges are $\pm\SI{25}{\newton}$ in $F_x$, $F_y$, and $F_z$, and $\pm\SI{125}{\milli\newton\meter}$ in $T_x$, $T_y$, and $T_z$, with a manufacturer-specified accuracy of $1\%$ of full scale, a crosstalk of $3\%$, and an overload capacity of $300\%$.
The sensor was paired with the Resense miniature evaluation electronics board (EVAL~100S-06-1, $54 \times 25 \times \SI{12.6}{\milli\meter}$), which digitises the strain gauge signals with a 24-bit ADC,
applies the manufacturer-provided calibration matrix onboard, and outputs calibrated forces (in newtons) and torques (in millinewton-metres) as seven 32-bit floating-point values per sample ($F_x$, $F_y$, $F_z$, $T_x$, $T_y$, $T_z$, and temperature).
The electronics was connected to the acquisition computer via its USB interface (\SI{12}{Mbit/s}) appearing as a virtual COM port, and configured at a sample rate of \SI{100}{\hertz}.
A custom Python script reads the 28-byte binary packets from the serial stream
and logs each sample with a wall-clock timestamp.
This sensor was used exclusively in subset~3 (dynamic motions with contact).

%%  --- ALIGNMENT AND CALIBRATION PROTOCOL ---
% To ensure consistency across all sensor modalities, every recording session follows a fixed preparation protocol.

First, the FBGS is aligned to the robot motion following the procedure.
Using a self-leveled laser (Bosh Professional GLL50-20) the robot is ensured straight on both the $x$ and $y$ axis, using cables to compensate enventual residual deformation of the backbone.
From this configuration, the ShapeCore software \textit{set\_straight} functionality allows us to update the fiber calibration matrix to comply with the registered shape.
Then, by pulling on the $+x$ tendon the robot is significantly bent in one direction, with usage of the laser the robot is ensured bent in the $xz$ plane, correcting any out of plane deformation with the remaining cables.
From this planar bent configuration, the \textit{set\_planar} functionality of ShapeCore ensures that the measurements are free from any residual twisting. 

Once the fiber is calibrated, two preliminary measures are performed, first one with all the tendon released measures the baseline noise on the sensors. 
Then, with the robot ensured in a straight configuration using the bosh laser, the configuration is recorded to provide a estimate of residual error in the FBGS and mocap measures.

Moreover, before each experiment, the laser is used to ensure that the robot is straight.
In this configuration, the active tendons are pre-tensioned to approximately \SI{1}{\newton} each, with a maximum difference between the two cables of an antagonistic pair $<$\SI{0.2}{\newton}.
This step minimizes the risk of cable slack, a well-known source of modeling error in tendon-driven continuum robots~\cite{Rucker2011a}.

% Second, the recording begins with the robot hanging straight down under gravity.
% This known reference configuration is used to verify the alignment between the OptiTrack rigid-body frames and the FBGS-reconstructed shape: any offset observed in the straight configuration is attributable to sensor misalignment rather than to robot deformation.

Finally, for every non-planar trajectory (\ie{} star, circular and Lissajous curves) the robot executes a calibration routine before the target motion.
The routine consists of a planar sweep in the $x$--$z$ plane (positive $x$ followed by negative $x$), then a planar sweep in the $y$--$z$ plane (negative $y$ followed by positive $y$).
Because aligning the FBGS fiber to the robot backbone with sub-degree accuracy is difficult in practice, a residual angular offset between the fiber deformation plane and the robot bending plane is expected.
The calibration sweeps characterize this offset, allowing it to be corrected in post-processing before the non-planar data is interpreted.

%%  --- ACQUISITION PIPELINE ---
\subsubsection{Acquisition pipeline}
For each experiment, a Windows batch script orchestrates the acquisition.
The script launches one independent process per sensor, each in its own terminal window.
Each sensor process polls its instrument at the native sampling rate (see the sensor specifications in Table \ref{tab:sensor_summary}), buffers all readings in memory, and writes them to a CSV file after the recording is complete.
The motor process is launched last, and waits \SI{2}{\second} before sending position commands, providing a sufficient delay to ensure all sensors to be already streaming before the actuators begin to move.
%% TODO: confirm the startup delay value. The original draft said "2 seconds";
%% the code shows time.sleep(1) for homing at the end, but the main control 
%% loop appears to start immediately after CAN bus init. Clarify whether there 
%% is an explicit sleep before the first motor command or just the natural 
%% startup latency.

Two sensors require dedicated proprietary software that runs independently from the
per-sensor Python acquisition scripts.
The FBGS fiber is managed by the ILLumiSense ShapeCore software (v1.1, FBGS),
which runs as a background process on the acquisition computer, connects to the
FBG-Scan~904 interrogator via USB, performs peak detection on the acquired spectra,
and reconstructs curvature, bending angle, and three-dimensional shape in real time.
ShapeCore streams its output over a local TCP connection.
A custom C++ client (\texttt{StreamClassTester}) connects to this stream,
receives the curvature, angle, and shape arrays at each time step,
timestamps each sample (using \texttt{std::chrono::high\_resolution\_clock}), and buffers all data in memory
until the recording is complete, at which point it writes a single CSV file.
%
The OptiTrack motion capture system is driven by Motive~3.4 (NaturalPoint, USA)
running on a dedicated Windows host PC.
Motive processes and solves the 3-D rigid-body poses at \SI{120}{\hertz} and
exposes them through the NatNet~4.4 SDK in unicast UDP mode.
A dedicated Python script on the acquisition machine receives each frame over a
point-to-point Ethernet link and timestamps it with the wall-clock of the
acquisition machine (\texttt{time.time()}), ensuring a homogeneous time base
across all sensor modalities.

%%  --- SYNCHRONIZATION ---
\subsubsection{Synchronization}
All timestamps in the recorded CSV files are therefore referred to the same operating-system clock, without the need for a hardware trigger.
The synchronization accuracy is bounded by the OS thread-scheduling jitter, on the order of \SI{1}{\milli\second} to \SI{2}{\milli\second} on the acquisition machine.


%%  --- POST-PROCESSING ---
\subsubsection{Post processing}
The raw sensor data is post-processed in MATLAB (\texttt{process\_data.m}).

% For the motion capture data, frames in which one or more markers are occluded are first discarded using the validity flag provided by the tracking software (a value of zero indicates occlusion).
The samples from the mocap system are converted from position ($x$, $y$, $z$) and quaternion to $\mathrm{SE}(3)$ poses.
Because these poses are expressed in the global motion-capture frame, they are re-expressed as transformations relative to the base of the robot.
To avoid propagating position and orientation error, we defined a base frame coincident with the position of disk\_0 and with axes aligned with the direction of the robot.
Then the pose of every disk frame is computed with respect to this common base.
%A static rigid-body transformation is then applied to orient the $z$-axis downward (the gravity direction) and to align the $x$- and $y$-axes with the bending-plane convention of the robot.

For the FBGS data, the interrogator software reconstructs the fiber shape with the arc-length parameter evolving along its internal $x$-axis, while the physical robot hangs along negative~$z$; therefore, a rigid rotation is applied to map the FBGS coordinate frame to the robot body frame.
Furthermore, because the fiber cannot be bonded to the backbone with sub-degree angular accuracy, a residual rotational offset exists between the fiber deformation plane and the true robot bending plane.
To characterize this offset, a singular-value decomposition is performed on the tip trajectory during the initial planar calibration sweep of each batch: the principal directions of the tip motion identify the bending-plane and the corresponding rotation angle, namely $\theta_{FBGS}$.
As the motion of the FBGS has to be aligned with the recorded motion of the mocap, a similar SVD-based procedure is applied on the tip motion as registered by the mocap, yielding $\theta_{mocap}$.
The correction applied to the FBGS, as a rotation along the z axis, of the amount $\theta_z = \theta_{FBGS} - \theta_{mocap}$.

All signals are then low-pass filtered with a fourth-order Butterworth filter (cutoff frequency \SI{20}{\hertz} applied in the forward and reverse directions (\texttt{filtfilt} in MATLAB) to achieve zero-phase distortion.
The filter's normalized cutoff is computed per signal using each sensor's native sampling rate, estimated from the median inter-sample interval.
The filtered signals are aligned in time by expressing all timestamps relative to the first actuator timestamp; because every sensor starts recording before the motors, these relative timestamps are initially negative.
All signals are then resampled onto a uniform \SI{100}{\hertz} grid spanning $t = 0$ to the end of the actuator recording using linear interpolation (\texttt{interp1}).
This operation simultaneously discards the sensor preamble (the negative-time samples recorded before the motors started) and brings all modalities to a common time base.

An independent geometric estimate of the cable displacement is derived from the motion-capture data to cross-validate the encoder-based measurements recorded by the actuators.
% The continuum robot is equipped with four antagonistic cables routed
% parallel to the elastic backbone and offset by $d = \SI{37.5}{mm}$ from its centreline along the $\pm x$ and $\pm y$ directions in the local cross-sectional frame.
% Five rigid spacer disks, instrumented with reflective markers, are tracked by the optical motion-capture system at $\SI{100}{Hz}$ and their six-degree-of-freedom poses are reconstructed relative to the robot base frame.
%Intrinsic $XYZ$ Euler angles are extracted from each disk's rotation matrix and phase-unwrapped over time to remove artificial discontinuities at the $\pm\pi$ boundary; the yaw component (rotation about the backbone axis) is subsequently forced to zero, consistent with the assumption that the robot undergoes purely spatial bending without torsion.

To obtain a more accurate description of the cable path, the five measured disk frames are interpolated to obtain the poses of the intermediate disks (not provided with mocap frames).
Positions are interpolated with a shape-preserving piecewise-cubic Hermite (PCHIP) scheme, while orientations are interpolated by converting each rotation matrix to a unit quaternion, applying PCHIP independently to each quaternion component, and normalizing the result to the unit sphere.
The consistency of the quaternion hemisphere is enforced between successive disks to avoid sign-flip artifacts.
At each interpolated frame the absolute position of every cable is computed by applying the local offset vector $\mathbf{d}_c$ through the interpolated rotation matrix.
The total length of cable $\ell_c$ is then approximated as the piecewise-linear sum of Euclidean distances between consecutive attachment points along the interpolated backbone.

The cable displacement is defined as the change in length relative to the straight (pre-motion) configuration, estimated by averaging the first $N_{\mathrm{ref}} = 10$ frames:
\begin{equation}
    \Delta \ell_c(t) \;=\; \ell_c(t) - \bar{\ell}_c^{\,\mathrm{ref}},
    \qquad
    \bar{\ell}_c^{\,\mathrm{ref}} = \frac{1}{N_{\mathrm{ref}}}
    \sum_{k=1}^{N_{\mathrm{ref}}} \ell_c(t_k).
\end{equation}


On the other hand, an independent estimate is obtained from the motor encoders as $\Delta \ell_c^{\,\mathrm{enc}}(t) = \bigl(\theta_c(t) - \bar{\theta}_c^{\,\mathrm{ref}}\bigr)\,r_s$, where $\theta_c$ is the measured shaft angle and $r_s$ is the spool radius (see Table \ref{tab:cables parameter}).
The two estimates are compared directly in the time domain; agreement between them constitutes a geometric consistency check, confirming that the encoder records and the motion-capture poses are mutually consistent and that the assumed cable-routing geometry is correct.

Cable tensions (measured with Mark10 sensors) and base wrench (measured with ATI-FT sensor) cannot be directly compared without a physical model relating forces to deformations.
To verify that these two families of measurements are mutually consistent, we employ a forward dynamics simulation based on the Geometric Variable Strain (GVS) approach~\cite{Boyer2021} as a cross-validation tool.

As the GVS model requires the parameters of the robot to be known, the next section will address how these parameters were roughly estimated.
Then, the following section discusses the comparison of the simulation results with the recorded data.

\subsubsection{Rod mechanical parameter characterization}
\label{sec:mech_char}
An equivalent bending rigidity $EI$ of the backbone was estimated with the quasi-static subset, where the robot was commanded to planar bending in the $xz$ and $yz$ planes.
% During each hold $w$, inline force gauges provided the tendon tension vector $\boldsymbol{\tau}_w$, while motion capture reconstructed the robot configuration from markers attached to a subset of spacer disks.

To avoid short settling effects after actuation, the first $\sim 200$ samples of each recording were removed and the remaining samples were averaged, yielding disk positions
$\{\mathbf{p}^{\mathrm{meas}}_i(w)\}_{i=1}^{N_{\mathrm{disks}}}$ in the base frame.
For a candidate value of $EI$, a static Cosserat-rod solver was used to predict the equilibrium shape under the measured tensions $\boldsymbol{\tau}_w$ by enforcing the boundary conditions through a root-finding routine.
The simulated disk positions evaluated at the same arc-length locations are denoted by $\mathbf{p}^{\mathrm{sim}}_i(EI,\boldsymbol{\tau}_w)$.

For each bending mode (plane), $EI$ was selected by minimizing the aggregated geometric discrepancy between simulated and measured disk positions over the corresponding set of trials $\mathcal{W}$:
\begin{equation}
\widehat{EI}
=
\arg\min_{EI}
\sum_{w \in \mathcal{W}}
\sum_{i=1}^{N_{\mathrm{disks}}}
\left\|
\mathbf{p}^{\mathrm{sim}}_i(EI,\boldsymbol{\tau}_w) - \mathbf{p}^{\mathrm{meas}}_i(w)
\right\|_2^2 .
\label{eq:EI_ident}
\end{equation}

The resulting plane-wise estimates were $\hat{EI}_{xz}=0.0925$ and $\hat{EI}_{yz}=0.08875$, a single value $\hat{EI}=0.092~\mathrm{N\cdot m^2}$ was used in the remainder of this work.

Then, the distributed weight of the rod is calculated from the measured weight of the robot ($0.102$ Kg) and the geometry of the rod (length $0.48$ m and radius $2$ mm) producing $\rho = 16910 \; \textbf{kg/m}^3$.

\subsection{Temporal synchronization}
\label{sec:synch_val}


All sensors are acquired as concurrent threads in a single Python process on a single machine.
Each thread timestamps its readings using the operating system wall-clock (\texttt{time.time()} in Python), ensuring that all sensors share the same clock source without requiring hardware synchronization.
The resulting synchronization error is bounded by the operating system thread scheduling jitter, which is on the order of \SI{1}{\milli\second} to \SI{2}{\milli\second} on the acquisition machine, well below the signal periods of interest (the highest dynamic content in the dataset is below \SI{20}{\hertz}).

\paragraph{Software requirements.}
The raw and processed data are stored as plain CSV files and can be loaded in any programming language.
The post-processing pipeline (\texttt{process\_data.m}) was developed and tested in MATLAB~R2025b, using only the built-in functions \texttt{butter}, \texttt{filtfilt}, and \texttt{interp1} from the Signal Processing Toolbox.
% The GVS simulation code (\texttt{test\_trajectories.m}) additionally requires the MATLAB ODE solvers.
Users who prefer Python can replicate the processing with \texttt{scipy.signal.butter}, \texttt{scipy.signal.filtfilt}, and \texttt{scipy.interpolate.interp1d}, or use \texttt{pandas.read\_csv} to load the data directly.

\paragraph{Known limitations and caveats.}
\begin{itemize}
    \item \textbf{FBGS coordinate alignment.}
    The FBGS fiber cannot be bonded to the backbone with sub-degree angular accuracy.
    A residual rotational offset between the fiber deformation plane and the robot bending plane is corrected in post-processing using the initial planar calibration sweeps (see Methods).
    Users working with the raw FBGS data must apply this correction themselves.
    \item \textbf{ATI-FT bias.}
    The ATI force/torque sensor exhibits a slowly drifting bias.
    A bias snapshot is recorded at rest before each experiment and subtracted from all readings.
    The unfiltered, bias-subtracted wrench is provided as \texttt{base\_wrench\_raw.csv}; the filtered version is \texttt{base\_wrench.csv}.
    \item \textbf{OptiTrack marker uncertanties.}
    Definition of frames with the OptiTrack markers suffer from uncertanties in the markers placement and assembly. 
    \item \textbf{Motor trajectory generation.}
    The motors are position-controlled; the trajectories are defined as sequences of waypoint angles with a maximum angular velocity limit.
    The actual tip trajectory depends on the robot dynamics and is not a perfect geometric curve.
\end{itemize}
